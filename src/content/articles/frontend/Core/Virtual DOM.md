React에서 UI 업데이트의 효율성을 극대화하기 위해 실제 DOM의 가벼운 복사본을 메모리 상에서 관리하는 기술.

DOM 조작이 느린 브라우저 환경에서 성능을 개선하는 데 필요하다.

## 왜 등장하였나?

Virtual DOM이 등장하기 전, 브라우저의 렌더링 과정은 다음과 같았다.

![alt text](<Pasted image 20260611090553.png>)

- **1. DOM & CSSOM 트리 생성**

  > 브라우저에서 HTML 파일을 읽고 [DOM](DOM) 트리를 생성한다.
  > DOM은 문서 요소를 트리 구조로 표현하며, 각 노드는 HTML 태그를 의미한다. 이 트리 구조를 `DOM 트리`라고 부른다.
  > HTML이 파싱되는 과정 중에 `<link>`나 `<style>`이 만나게 되면 CSS 파일을 파싱하여 `CSSOM 트리`를 생성한다.
  > CSSOM은 CSS를 파싱하여 트리 구조로 나타낸 객체 모델로, HTML 문서의 각 요소에 적용된 CSS 스타일 규칙을 나타내며, DOM과 결합하여 화면에 표현될 준비를 한다.

- **2. 렌더 트리 형성**

  > DOM과 CSSOM을 결합하여 렌더 트리를 생성한다.
  > 렌더 트리는 화면에 표시될 요소들만 가지고 있으며, 각 요소에 적용될 스타일과 위치에 대한 정보를 담고 있다.

- **3. 레이아웃**

  > 렌더 트리를 기반으로 각 노드의 화면에서의 정확한 위치(좌표)와 크기를 계산한다.
  > 이미 계산된 스타일 값을 바탕으로 각 요소의 기하학적 정보(위치, 크기, 박스 모델)를 계산한다.

- **4. 페인트**

  > 레이아웃 단계에서 계산된 정보를 바탕으로 렌더 트리의 각 노드를 실제 픽셀(색상, 텍스트, 이미지, 그림자 등)로 그린다.
  > 페인팅 성능을 높이기 위해 요소를 여러 개의 레이어(Layer)로 분리하여 그리기도 한다.

- **5. 컴포지팅**
  > 페인트에서 그린 여러 레이어 요소들을 올바른 순서로 합성하는 단계이다.
  > 브라우저는 올바른 순서로 합성하여 화면에 정확히 렌더링한다.
  > 만약 z-index나 position이 곂치는 경우, 겹침 순서를 정확히 처리한다.

이처럼 브라우저는 복잡한 렌더링 과정을 통해 화면을 그린다.

React는 SPA 기반이다. 한 페이지에서 렌더링 이후의 추가 렌더링 작업이 빈번하게 일어나는데, DOM 변경마다 `리플로우`와 `리페인팅` 같은 비싼 렌더링 과정이 반복된다.
이를 해결하기 위해 React는 Virtual DOM을 사용함으로서, 변경된 부분만 실제 DOM에 반영함으로서, 이 비용을 줄일 수 있다.

## Virtual DOM VS DOM

| Virtual DOM                            | 실제 DOM                                        |
| -------------------------------------- | ----------------------------------------------- |
| 원본 DOM의 가벼운 복사본               | HTML 요소들을 트리 구조로 표현한 것             |
| JavaScript 라이브러리에 의해 유지 관리 | HTML 요소 파싱 -> 브라우저에 의해 유지 관리     |
| 조작 후 변경된 요소들만 리렌더링       | 조작 후 전체 DOM을 리렌더링                     |
| 가벼운 업데이트                        | 무거운 업데이트                                 |
| 빠른 성능과 최적화된 UX                | 느린 성능과 품질이 떨어지는 UX                  |
| 배치 업데이트를 수행하여 매우 효율적   | 각각의 업데이트 이후 DOM을 리렌더링 -> 비효율적 |

## Virtual DOM이 더 빠른가?

반은 맞고 반은 틀린 얘기다.

Virtual DOM은 생성된 가상 돔을 실제 DOM과 비교하는 과정이 있기 때문에, 실제 DOM을 조작하는 것보다 빠르다고 단정할 순 없다.
하지만 더 빠르다고 할 수 있는 이유는 최적화를 통해 실제 DOM 조작을 최소화하여 필요한 부분만 DOM을 수정하기 때문이다.

그렇다면 Virtual DOM이 무조건 좋은 것일까?

만약 규모가 작고 정적인 웹 애플리케이션이라면, 일반 DOM의 성능이 더 좋을 것이다.
그러나, 지금의 React는 동적인 웹 애플리케이션에 중점을 두고 있다.

[[SPA]]에서는 DOM 조작이 빈번하게 발생한다.
변화를 적용하기 위해서는 브라우저가 많은 연산을 하게 되고, 결국 전체적인 프로세스가 비효율적이게 된다.

## Virtual DOM의 동작 과정

과정은 크게 3단계로 나뉜다.

- **1. 가상 돔 생성**

  > 변화된 부분을 기반으로 새로운 가상 돔 트리 구조를 메모리에 생성한다.
  > 이는 자바스크립트 객체 형태이며, 화면에 직접 그려지지 않으므로 생성 속도가 매우 빠르다.

- **2. 비교 알고리즘**

  > 현재 Virtual DOM과 이전 Virtual DOM의 상태를 비교하여 차이를 계산한다.

- **3. 변경된 부분 반영**
  > 변경된 노드들의 리스트를 실제 DOM에 반영한다. 이를 통해 브라우저는 DOM 조작으로 인한 레이아웃 계산을 한 번만 할 수 있다.
